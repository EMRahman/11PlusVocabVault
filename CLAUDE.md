# CLAUDE.md

Guidance for AI agents working on **11+ Vocab Vault**. This codebase is
developed and maintained **exclusively by AI** — optimize for machine
verifiability and low per-session context cost, not human aesthetics. Keep this
file current: when you change architecture, update the relevant section here in
the same PR.

## What this is

A zero-build, zero-dependency **static web app** (vanilla HTML/CSS/ES modules)
that helps 8–11 year-olds learn vocabulary for UK 11+ exams. Hosted on GitHub
Pages. There is **no bundler, no transpile, no runtime dependency** — the files
served are the files in the repo. Keep it that way unless explicitly told
otherwise.

## Verify your changes (do this before every PR)

```bash
node --test            # full suite (Node's built-in runner; no install). CI runs this on Node 20.
node --check js/app.js # syntax-check any file you edited (esp. the big ones)
```

- There is **no browser/DOM test harness**. `node --test` covers only pure
  logic in the small ES modules. For DOM-heavy changes (the "modes"), verify by
  *structural equivalence* and `node --check`; when you extract logic, make it
  pure and add characterization tests (see `test/selection.test.js` as the
  model). A throwaway stubbed-DOM harness is a reasonable way to smoke-test a
  factory before pushing.
- CI = `.github/workflows/test.yml` (just `node --test`). A second workflow,
  `update-build-info.yml`, runs on push to `main` (see Gotchas).

## Test & CI governance

The tests are the **only** automated safety net — there are no browser/DOM tests
and you can't run the app, so `node --test` is the sole check that a change is
correct. Treat `test/` and `.github/workflows/` as load-bearing.

- **Adding or strengthening tests is always welcome — no approval needed.** When
  you fix a bug or add behaviour, add coverage in the same PR.
- **Never weaken the safety net to get a green build.** Do not delete tests, skip
  or disable them (`test.skip`, `.only`, `.todo`, `skip: true`, commenting out,
  early `return`), loosen or remove assertions, relax the `data-integrity`
  checks, or disable/relax CI in `.github/workflows/`. Each of these needs
  **explicit human approval** and must be called out prominently in the PR
  description with the reason.
- **If a test fails, fix the code, not the test.** If a test is genuinely wrong
  or obsolete, don't quietly change it — flag it in the PR and get explicit human
  sign-off, with the reasoning.
- `test/coverage-floor.test.js` enforces this mechanically: it fails if the
  test/assertion count drops below a committed baseline, or if any test is
  skipped. Lowering a baseline there is itself a weakening change — same approval
  rule applies.

### Data files (`data/*.json`) — top-level key each mode reads

| File | Key | File | Key |
|------|-----|------|-----|
| `words.json` | `words` | `proverbs.json` | `collections` |

| `history.json` | `articles` | `animals.json` | `animals` |
| `insects.json` | `insects` | `fables.json` | `fables` |
| `space.json` | `space` | | |
| `word-positions.json` | `positions`,… (Word Universe layout) | `animal-constellations.json` | `segments` |
| `word-explorer.json` | `mood`,`etymology`,… | | |

Word schema (`words.json`): `word, word_type, pronunciation, definition,
sentence_usage, synonyms[], antonyms[], usefulness_rating (1–5)`, plus a
pipeline-baked `themed_quest` used by Story Quest (its `theme`, `word`, and
pre-blanked `sentence` are required; `synonym`/`antonym` are optional).

**Multiple meanings.** A polysemous word (e.g. *Objective* = adjective + noun) may
also carry an OPTIONAL `meanings[]` array — each sense is `{ word_type, definition,
sentence_usage, synonyms[], antonyms[] }` (plus an OPTIONAL `pronunciation` for a
heteronym whose sense is said differently, e.g. the noun *abuse* `uh-BYOOS` vs the
verb `uh-BYOOZ`), primary first. `meanings[0]` mirrors the flat fields exactly (the
"mirror invariant"), so single-sense / legacy words can
omit `meanings[]` entirely and every consumer reads one shape via `js/meanings.js`
`getMeanings()`. **Display surfaces** (word card, modal "Other meanings",
Flash-Blitz) show all senses; **quizzes/games and the visualisations stay on the
primary (flat) sense.** Extra senses are generated by the
`scripts/build-meanings-prompts.js` → `merge-meanings.js` Haiku pipeline (keeps the
curated primary, appends validated extras, caps at 3). See
`MULTIPLE_MEANINGS_PLAN.md`.

The word schema, the OPTIONAL `meanings[]` array, the `themed_quest` payload, and
every content-collection key are enforced by `test/data-integrity.test.js` —
**update that test when you add a data file or collection.**

## Key patterns (read before editing app.js)

- **State singletons** (`store.js`): mutate object *properties* in place; **never
  reassign** the imported binding (`mastery = {...}` throws — ES module imports
  are read-only live bindings). `storage.js` load fns clear-and-repopulate.
- **"Modes" are overlays.** Each mode = a launch button + a full-screen overlay
  with a library screen and a reading/play screen (markup in `index.html`).
  Every mode has an `init<Mode>()` wired in **`loadWordData()`'s `.then`** (the
  authoritative list of active modes). Modes load their content via `fetch()`
  and re-render if their overlay is already open.
- **Reading modes share a factory.** History/Animals/Insects/Fable are built by
  `createReadingMode(config)` in app.js (config = prefix, progressKey, dataFile,
  dataKey, returnTo, itemNoun, optional `subtitleField`/`moralField`,
  loadingMessage). DOM ids follow `<prefix>-...`. Proverbs is **not** folded in
  (culture picker + 3 screens + native-script cards). To add a reading mode: add
  `index.html` markup with `<prefix>-*` ids, then one `createReadingMode({...})`.
- **Quiz return routing.** A scoped quiz carries `returnTo`; `closeQuizOverlay()`
  sends the user back via the **`readingReturnHandlers` registry** (factory modes
  register themselves) with explicit branches for `story`/`news`/`proverbs`.
- **Visualisation integration is via `window.*` globals.** app.js calls
  `wireWordUniverse`/`wireConstellationQuest`/`wireExplorerExtras`, which invoke
  `window.initWordUniverse`, `window.initMoodMap`, etc. (guarded by `typeof`).
  app.js exposes `window.vaultRecordAnswer` so the constellation game feeds
  mastery. (Inconsistency: some viz files are ES modules, some are global
  scripts — see Gotchas.)
- **Persistence:** all progress is in `localStorage` under `vocabVault_*` and
  `11plus-tts-*` keys (quizBest(s), questProgress, story/history/animals/insects/
  fable/proverbsProgress, dailyNews, detective/scramble/snapBests, tts-voice/pitch).
  Changing a key silently wipes users' saved progress — don't rename casually.

## Common tasks

- **Add words:** append to `data/words.json` `words[]` (full schema). Integrity
  test enforces non-empty fields, rating 1–5, unique names. `meanings[]` is
  optional — omit it for a single-sense word.
- **Add another meaning to a word:** add an OPTIONAL `meanings[]` array whose `[0]`
  mirrors the flat fields and whose later entries are the extra senses
  (`{ word_type, definition, sentence_usage, synonyms[], antonyms[] }`). The bulk
  path is `node scripts/build-meanings-prompts.js` → paste to Haiku → `node
  scripts/merge-meanings.js`. The integrity test enforces the mirror invariant and
  distinct senses; display surfaces pick them up automatically via `getMeanings()`.
- **Add a content collection / reading mode:** new `data/<x>.json` → add to the
  `collections` map in `test/data-integrity.test.js` → add markup + a
  `createReadingMode({...})` (or a bespoke init for non-reading shapes).
- **Extract/refactor logic out of app.js:** move *pure* logic to a small ES
  module (state passed in as args, like `selection.js`), import it, and add
  characterization tests with golden values. This is the preferred way to make
  app.js logic testable and to shrink per-session context.

## Gotchas / coupling hazards (AIs trip on these)

1. **`data-integrity.test.js` must track new data files** — it's the only guard
   on the JSON the app depends on.
2. **Cache-busting is entry-only.** `update-build-info.yml` rewrites `?v=` on the
   `<script>` tags in `index.html` but **deliberately does NOT stamp ES-module
   import specifiers** inside `js/*.js` (stamping breaks module identity under
   Node's test runner). Never add `?v=` to `import` paths.
3. **Theme generation is duplicated.** `scripts/themes-lib.js` (CommonJS) bakes
   each word's `themed_quest` into `words.json` and defines the `WORLDS` list;
   the Story Quest worlds (ids/order) are mirrored in app.js's quest state. The
   `themes-lib.js` header comment about "getWordTheme in app.js" is **stale**
   (app.js now reads the baked theme, not a live `getWordTheme`). If you touch
   `WORLDS`, keep both sides in sync and regenerate `words.json` via `scripts/`.
4. **ESM vs CommonJS split:** root `package.json` is `"type":"module"` (app +
   tests are ESM). `scripts/` has its own `package.json` pinning CommonJS. Don't
   mix.
5. **Mixed viz loading:** `word-universe`/`word-quest-3d` load as `type="module"`;
   `mood-map`/`word-portrait`/`word-roots-garden`/`animal-constellation` load as
   global scripts and expose `window.init*`. If you add a viz, match the global
   pattern or update `index.html` + the `wire*` glue in app.js accordingly.
6. **No DOM tests** — passing `node --test` does NOT exercise `app.js`. Don't
   assume green CI means a mode still works; reason about DOM/id correctness
   explicitly.

moved to `data/comics.json`; History/Animals/Insects/Fable unified into
`createReadingMode`; daily-news/weakest-words logic extracted to tested
